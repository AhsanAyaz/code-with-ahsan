# Milestone v7.0: Agentic Orchestra Upgrade

**Status:** 🟡 In planning (started 2026-05-22)
**Phases:** 3 (Phases 06 → 08 in `.planning/phases/`)

## Why this milestone

`community_assistant` Discord bot (production on Cloud Run since 2026-04 / Phase 02-adk + Phase 02.1-adk-content-external-knowledge) ships as 1 root `LlmAgent` + 6 sub_agents using LLM-driven transfer only. Self-audit (CFP framing doc, 2026-05-22) measured it at ~3/10 against the speaker's own "Zero to Agentic Orchestra with Google ADK" talk material.

Gaps grouped into three structural levels:

1. **Topology** — no `ParallelAgent`/`SequentialAgent`/`LoopAgent`, no `output_key` whiteboard, `featured_resources` is a Python dict instead of an `AgentTool`-wrapped specialist.
2. **Safety + observability** — zero callbacks. No `before_model_callback` PII redaction, no `before_tool_callback` cache, ad-hoc print logging instead of structured lifecycle callbacks.
3. **Quality + memory** — no critic/refinement loops, no clarification loop for mentor/project intake, no persistent session (InMemorySessionService is wiped on Cloud Run cold start).

Production signal: real measured `latency_ms=18933` (~19s) on a 3-source `external_knowledge` query (logged in v6.0 STATE.md). Sequential httpx fan-out inside one LlmAgent is the primary culprit.

This milestone closes the 10-pattern scorecard AND produces the live-demo backbone for the talk.

## Phases

### Phase 6: Agent Workflow Refactor + State Wiring

**Goal:** Transform `community_assistant` from a flat coordinator-of-6 into a topology that uses ADK workflow primitives correctly — `ParallelAgent` for independent fan-out, `output_key` whiteboard for cross-agent context, `AgentTool` for synchronous specialist calls — without touching callbacks or session services yet.

**Depends on:** Phase 02 (community_assistant base), Phase 02.1 (content/external sub-agents)

**Requirements:**
- `AGENT-PAR-01` — `external_knowledge_agent` refactored from 1 LlmAgent with 3 sequential httpx tools into a `ParallelAgent` of 3 specialist LlmAgents (`github_researcher`, `devto_researcher`, `so_researcher`), gathered by a downstream synthesizer LlmAgent inside a wrapping `SequentialAgent`. Each specialist writes to a distinct `output_key` (`gh_result`, `devto_result`, `so_result`). The synthesizer reads `{gh_result} {devto_result} {so_result}` via instruction templating.
- `AGENT-PAR-02` — P95 latency on 3-source queries drops by ≥40% (target: <12s from current ~19s).
- `AGENT-STATE-01` — `onboarding_agent` writes `user_skill_level` and `user_goals` to session state via `output_key`.
- `AGENT-STATE-02` — `mentorship_agent`, `projects_agent`, `roadmap_agent` instructions read `{user_skill_level?}` and `{user_goals?}` (optional templating — `?` because first turn won't have them).
- `AGENT-TOOL-01` — `featured_resources.py` Python dict promoted to `featured_resources_agent` LlmAgent, wrapped in `AgentTool`, and called from `content_agent.tools=[...]` instead of being prepended by Python code.
- `AGENT-TEST-01` — pytest coverage added for the new topology: parallel fan-out (mock 3 httpx clients, assert all 3 called within one event loop tick), state propagation (assert downstream agent reads upstream `output_key`), AgentTool invocation (assert content_agent calls featured_resources via tool path, not Python prepend).

**Success Criteria (what must be TRUE):**
1. Running `adk web` shows `external_knowledge` queries firing 3 branches interleaved in Events tab (not stacked sequentially).
2. State tab populates `user_skill_level` / `user_goals` after first onboarding turn; subsequent mentor/project/roadmap turns reference those keys.
3. Featured-resource items appear in `content_agent` output via tool call (visible in Events tab), not via Python list-prepend.
4. Production Cloud Run deploy survives 24h with the new topology — no regression on Discord HMAC-hashed usage metrics dashboard.
5. Talk demo `parallel_research` slide can be live-swapped for `external_knowledge` running against production — same fan-out pattern, real production code.

**UI hint:** no (backend/agent only; no Next.js page changes).

### Phase 7: Agent Safety + Observability Callbacks

**Goal:** Add the three callback layers that the talk's "guardrails + observability plane" beat requires — PII redaction at the model layer, tool-result caching at the tool layer, structured lifecycle logging at the agent layer — replacing the ad-hoc `print` statements and Next.js-only ISR.

**Depends on:** Phase 6 (callbacks attach to the refactored agent tree)

**Requirements:**
- `AGENT-CB-MODEL-01` — `before_model_callback` PII sanitizer pattern (per `ai-agents-google-adk/7-agents-and-callbacks/example_02`) attached to root_agent. Redacts credit-card patterns, email-like patterns, and Discord IDs before LLM sees them. Logged at `INFO` with sanitization count, redaction type only — never raw values.
- `AGENT-CB-TOOL-01` — `before_tool_callback` cache + `after_tool_callback` cache-write pattern (per `example_05`) applied to `search_blog_posts`, `search_youtube_videos`. Cache stored in `tool_context.state["app_cache"]` (session-scoped per ADK semantics). TTL 600s. Cache key: `{tool_name}_{normalized_query}_{locale?}`.
- `AGENT-CB-TOOL-02` — Cache hit rate logged via Cloud Logging structured payload (`event: "tool_cache_hit", tool: "<name>", age_s: <int>`). Aim: ≥30% hit rate on `search_blog_posts` within 7 days of deploy.
- `AGENT-CB-AGENT-01` — `before_agent_callback` + `after_agent_callback` lifecycle logger (per `example_01`) attached to every sub-agent. Emits structured JSON: `{event: "agent.enter"|"agent.exit", agent: "<name>", duration_ms: <int>, session_id_hash: <hmac>}`. Replaces existing scattered `print` statements.
- `AGENT-CB-AGENT-02` — Discord bot stdout still emits HMAC-hashed user_id metric (preserved from v6.0 decision). Callbacks do not weaken PII story — raw user content never appears in logs.
- `AGENT-TEST-02` — pytest cases for: PII redaction (positive + negative samples), cache hit/miss (warm/cold), cache TTL expiry, lifecycle callback emission.

**Success Criteria (what must be TRUE):**
1. Sending a query containing a fake credit-card pattern (e.g., `4111-1111-1111-1111`) into Discord shows `[REDACTED PII]` in the LLM prompt as visible in adk web's `before_model_callback` debug output; the raw number never appears in any log.
2. Asking the same blog question twice within 600s the second response is served from `tool_context.state` cache (visible in Cloud Logging with `tool_cache_hit` event).
3. Cloud Logging shows per-sub-agent `agent.enter` / `agent.exit` events with `duration_ms` for every Discord turn — usable for a P95-per-agent dashboard.
4. No regression on existing usage metric pipeline (HMAC-hashed user_id + query_topic + query_len + latency_ms still emitted).
5. Talk demo can stop on stage and click `before_model_callback` debug pane to show PII redaction live; click `before_tool_callback` debug pane to show cache hit timestamp.

**UI hint:** no.

### Phase 8: Agent Quality + Memory (Loops + Sessions)

**Goal:** Add the `LoopAgent` quality + interaction primitives that the talk teaches but production lacks — a critic loop for mentor/project recommendation quality, a clarifier loop for intake-before-recommend, and a Firestore-backed `DatabaseSessionService` so the bot remembers conversation across redeploys.

**Depends on:** Phase 7 (lifecycle callbacks needed to instrument loop iterations cleanly)

**Requirements:**
- `AGENT-LOOP-CRITIC-01` — `mentor_critic` LoopAgent wraps `mentorship_agent` recommendations: critic LlmAgent reads candidate list + user intake state, either calls `exit_loop` tool (recommendations are relevant given known intake) OR signals "widen filters / re-search". `max_iterations=2`.
- `AGENT-LOOP-CRITIC-02` — Same critic pattern for `projects_agent` ("are these projects matched to user goals?").
- `AGENT-LOOP-CLARIFY-01` — `mentor_intake_clarifier` LoopAgent runs BEFORE mentor recommendation. Clarifier LlmAgent reads session state (`user_skill_level?`, `user_goals?`, `user_timezone?`, `user_focus_area?`), decides if intake is complete enough to recommend, OR generates next clarifying question + writes question to `next_clarifying_question` state and exits the loop (control returns to user, next user turn writes their answer to state, next loop iteration re-evaluates). `max_iterations=3` per user-turn (prevents runaway questioning in a single turn).
- `AGENT-LOOP-CLARIFY-02` — Same clarifier pattern for `projects_agent` (community-project contribution path): clarify skill level + time commitment + tech preference before recommending projects to contribute to.
- `AGENT-SESSION-01` — Swap `InMemorySessionService` → `DatabaseSessionService` backed by Firestore. Session docs live at `agent_sessions/{user_id_hmac}/turns/{turn_id}` (HMAC keyed by `USAGE_HASH_SECRET` — same hash convention as v6.0 logging decision so raw Discord user_id never lands in Firestore).
- `AGENT-SESSION-02` — Sessions survive Cloud Run cold start + redeploy. User can ask "what was my goal again?" 30 minutes after a redeploy and the bot remembers.
- `AGENT-SESSION-03` — Session retention policy: prune `agent_sessions/*/turns/*` older than 30 days via existing `.github/workflows/ambassador-activity-checks.yml`-style daily cron (new workflow + `scripts/agent-session-prune.ts`).
- `AGENT-TEST-03` — pytest cases for: critic loop exits on relevant recommendations, critic loop widens filters when irrelevant, clarifier asks ≤3 questions per turn, session persistence across simulated cold start.

**Success Criteria (what must be TRUE):**
1. User asks "find me a mentor" with empty session state; clarifier asks ≥1 question (probably skill level), pauses for user reply, then on the next turn either asks one more or proceeds to recommendation.
2. Critic loop runs after `mentorship_agent` returns candidates; visible in Events tab as a sub-loop with `Critic → exit_loop` on iteration 1 or 2.
3. Bot remembers user goals after Cloud Run redeploy: simulate with `gcloud run services update --revision-suffix=test` and verify session continuity.
4. `agent_sessions/{hmac}` documents in Firestore: turns sub-collection populated, raw Discord user_id absent everywhere (only HMAC).
5. Daily cron successfully prunes turns >30d old without dropping active sessions (idempotent + safe).
6. Talk demo `content_orchestra` LoopAgent slide can live-swap to `mentor_clarifier` — same `exit_loop` tool pattern, real user-facing flow.

**UI hint:** no.

## Milestone-Level Decisions

- **3-phase split rather than 1 fat phase** (user decision 2026-05-22). Reasoning: structural (phase 6) must land before callbacks (phase 7) so the callbacks attach to the final tree; sessions (phase 8) needs lifecycle callbacks (phase 7) for clean per-loop-iteration logging.
- **Talk-first framing.** Each phase's success criteria includes a "talk demo can live-swap" line. The refactor is judged in part by whether it produces stage-ready demos.
- **Privacy continuity from v6.0.** HMAC-hashed user_id (USAGE_HASH_SECRET) is preserved across all 3 phases. No raw user content in logs, no raw Discord IDs in Firestore session docs.
- **Cron convention preserved.** Phase 8 session pruning uses GitHub Actions + tsx scripts, per CLAUDE.md/STATE.md standing rule ("Cron jobs — always use GitHub Actions + scripts/").
- **MCP toolsets + hierarchical decomposition out of scope.** Namedrop in talk only.

## Cross-Phase Contracts

- `output_key` namespace lock (established in Phase 6, used by all later phases):
  - `user_skill_level` (string, written by `onboarding_agent`)
  - `user_goals` (string, written by `onboarding_agent`)
  - `user_timezone` (string, written by `mentor_intake_clarifier` if asked)
  - `user_focus_area` (string, written by `mentor_intake_clarifier` if asked)
  - `gh_result` / `devto_result` / `so_result` (external_knowledge fan-out outputs)
  - `next_clarifying_question` (string, written by clarifier when more intake needed)
- HMAC convention: every cross-turn identifier hashed with `USAGE_HASH_SECRET` before persistence — preserved from v6.0.
- Cloud Logging structured event names (allocated for Phase 7+):
  - `agent.enter` / `agent.exit`
  - `tool_cache_hit` / `tool_cache_miss`
  - `pii_redacted` (with type + count, never raw values)
  - `loop_exit` (with agent name + iteration)
