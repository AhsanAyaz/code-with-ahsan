## Spacing Scale

Declared values (must be multiples of 4). Source: DaisyUI/Tailwind defaults + `.page-padding` utility observed in existing pages.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, inline label gaps (`gap-1`) |
| sm | 8px | Compact element spacing, form field internals (`gap-2`) |
| md | 16px | Default element spacing, card body gaps (`gap-4`) |
| lg | 24px | Section padding, card body outer (`gap-6`) |
| xl | 32px | Layout gaps between major sections (`gap-8`) |
| 2xl | 48px | Major section breaks (`space-y-12`) |
| 3xl | 64px | Page-level spacing (reserved for dashboard Phase 5) |

Exceptions:
- Touch targets for mobile action buttons (flag, strike increment): minimum 44px height — use `btn btn-sm min-h-[44px]` on mobile viewports
- Loading spinner containers: `py-12` (48px) vertical padding — matches ApplyWizard pattern

## Typography

Source: Rubik font stack confirmed in tailwind.config.js and globals.css. DaisyUI utility classes map to these sizes. Confirmed from existing component inspection (DecisionDialog, AmbassadorPublicCardSection, ApplicationsList).

| Role | Size | Weight | Line Height | DaisyUI Class |
|------|------|--------|-------------|---------------|
| Body | 16px | 400 (regular) | 1.5 | base prose or none |
| Label | 14px | 700 (bold) | 1.4 | `label-text font-bold` |
| Heading | 20px | 700 (bold) | 1.2 | `card-title` / `text-xl font-bold` |
| Display | 28px | 700 (bold) | 1.2 | `text-2xl font-bold` (page h1 headings) |

Maximum 2 weights in use per surface: regular (400) for body/help text and bold (700) for labels, headings, and card-titles. Semibold (600) is not used anywhere in Phase 4.

## Color

Source: globals.css `:root` overrides. DaisyUI semantic tokens are the contract — do not use raw hex values in new components.

| Role | Value | DaisyUI Token | Usage |
|------|-------|---------------|-------|
| Dominant (60%) | base-100 (light) / base-100 (dark) | `bg-base-100` | Page background, card surfaces |
| Secondary (30%) | base-200 / base-300 | `bg-base-200`, `bg-base-300` | Admin table rows, sidebar, filter bars |
| Primary accent (10%) | #8f27e0 (purple) | `text-primary`, `btn-primary`, `badge-primary` | Primary CTA buttons, active state indicators, referral code display |
| Success | DaisyUI success | `badge-success`, `alert-success` | Accepted status, on-time report badge, strike-free state |
| Warning | DaisyUI warning | `badge-warning`, `alert-warning` | Under-review status, overdue report badge, cron-flagged items |
| Error / Destructive | DaisyUI error | `badge-error`, `alert-error`, `btn-error` | Strike increment confirm, declined status, flagged/hidden events |
| Info | DaisyUI info | `alert-info`, `badge-info` | Empty states, neutral notices (no active cohort, etc.) |

Accent (primary) reserved for:
- Primary CTA buttons only (Submit Report, Log Event, Copy Referral Code)
- Referral code display chip (visually distinct, copyable)
- Active tab/filter indicator on admin members page

Destructive (error) reserved for:
- Strike increment confirmation button
- Event flag/hide action (admin only)
- "Confirm strike" modal confirm button

Do NOT use `btn-primary` for secondary navigation actions — use `btn-ghost` or `btn-outline` per existing patterns.

## Interaction States

Every interactive element must implement all four states. No exceptions.

| State | Implementation |
|-------|---------------|
| Default | DaisyUI component default |
| Loading | `loading loading-spinner loading-sm` inside button; button `disabled` |
| Success | `toast.success(...)` via ToastContext (existing pattern) |
| Error | `alert alert-error` inline OR `toast.error(...)` for non-blocking errors |
| Empty | `alert alert-info` with descriptive copy + next-step link |
| Disabled | `disabled` attribute + `opacity-50` via DaisyUI |

## Visual Hierarchy Notes

### `/ambassadors/report`

- Focal point: `MonthlyReportForm` is the primary action surface. Position it above `EventList` so the ambassador's attention lands on the report first.
- `ReportStatusBadge` renders inline beside the page heading to immediately communicate submission state without requiring scroll.
- `LogEventForm` is subordinate — render it below the report form or behind an expand/add button to avoid visual competition.

### `/admin/ambassadors/members/[uid]`

- Focal point: `ActivitySummaryPanel` stat grid sits at the top of the right column, establishing context before the admin takes action.
- `StrikePanel` and `CronFlagsPanel` follow below the stat grid — never above. An admin must see the evidence (flags) before the action (strike button).
- `StrikeConfirmModal` must display `{displayName}` prominently in the heading — prevents confirming the wrong ambassador record.

## Accessibility Constraints

- All form fields require a visible `<label>` with `htmlFor` matching input `id` — matches existing pattern in AmbassadorPublicCardSection
- All modal dialogs use `<dialog open className="modal modal-open">` — matches DecisionDialog pattern
- All destructive actions require a confirmation modal before executing — no single-click destructive mutations
- Strike increment confirmation modal must display the ambassador's display name in the heading copy — prevents admin confirming the wrong record
- Copy-to-clipboard on referral code: provide visible feedback (`toast.success` or inline "Copied") within 500ms
- Loading spinners inside buttons must disable the button simultaneously — matches existing `disabled={busy}` pattern
