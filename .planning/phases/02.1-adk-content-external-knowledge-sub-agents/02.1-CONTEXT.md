# Phase 2.1: ADK Content & External Knowledge Sub-Agents - Context

**Gathered:** 2026-05-17
**Status:** Ready for planning
**Source:** Direct user direction (Discord community feedback + design conversation 2026-05-17)

<domain>
## Phase Boundary

Extend the existing ADK community assistant (`agent/community_assistant/`) with three new sub-agents:

1. **`content_agent`** — surfaces Ahsan's owned content (Ghost blog at blog.codewithahsan.dev, YouTube channel)
2. **`external_knowledge_agent`** — surfaces third-party developer knowledge (GitHub, dev.to, Stack Overflow)

Plus the supporting Next.js API surface that wraps Ghost + YouTube APIs with ISR caching (so the ADK agent does not hit upstream APIs on every Discord message and blow daily quotas).

**In scope:**
- New Python sub-agents in `agent/community_assistant/sub_agents/`
- New Next.js API routes under `src/app/api/content/...` for blog/YT (proxy + cache via Next.js `revalidate`)
- New `platform_client.py` functions to call those routes (consistent with existing `fetch_mentors`, `fetch_projects` pattern)
- Direct API calls from Python for GitHub + dev.to + Stack Overflow (no caching layer — those have generous quotas and rate-limit gracefully)
- Root agent (`community_assistant/agent.py`) routing updates to include the new sub-agents
- Tests for new tools (Python `agent/tests/` + Vitest for Next.js routes)
- ADK redeploy

**Out of scope (deferred):**
- Migrating blog/YT content to Firestore + vector embeddings for semantic search. This is the long-term direction (matches the existing `mentorship_profiles.bioEmbedding` pattern) but is deferred to a future phase — ISR caching is enough for the demo.
- Discord-specific UX (rich embeds, attachments). Out of ADK agent scope.
- Other third-party sources beyond GH + dev.to + Stack Overflow (e.g., HN, Reddit, Twitter/X) — explicitly excluded per user direction "that's it".

<decisions>
## Implementation Decisions

### Sub-Agent Topology
- **Two new sub-agents, not three.** Owned content (blog + YT) lives in one `content_agent`; third-party sources live in one `external_knowledge_agent`. Splitting by trust level (Ahsan's owned vs. external) — different prompting + citation needs.
- Each sub-agent uses `google.adk.agents.LlmAgent` matching existing pattern (`mentorship_agent`, `projects_agent`, `roadmap_agent`).
- Model: `gemini-2.5-flash` (matches existing sub-agents).

### Caching Strategy
- **Ghost blog + YouTube:** Proxy through Next.js API routes with ISR cache (`revalidate: 3600` = 1hr). Pattern mirrors existing `platform_client.py` design (Python tools call Next.js endpoints; cache + business logic stays in Next.js).
  - `/api/content/blog/search?q=` — Ghost Content API
  - `/api/content/youtube/search?q=` — YouTube Data API v3
- **GitHub + dev.to + Stack Overflow:** Direct API calls from Python. Quotas are generous (GH unauth = 60/hr; dev.to ~1000/hr; SO ~10k/day unauth) for our usage profile. Add API keys/PATs if env vars present for higher quotas, otherwise unauth.
- Caching deferred to ISR layer for now. Firestore-backed catalog + vector search is the future migration path but explicitly out of scope.

### Sub-Agent Routing (Root LlmAgent Instructions)
- Root agent (`community_assistant/agent.py`) gets two new bullets in its delegation rules:
  - `content_agent`: blog posts, YouTube videos, courses Ahsan has published
  - `external_knowledge_agent`: trending repos, recent articles, technical Q&A from outside the community
- Routing must prefer existing specialized sub-agents (mentorship/projects/roadmaps) when query matches their domain. External-knowledge agent is a fallback, not a default.

### Source Allowlist (External Knowledge)
- **GitHub:** Search repos + recent releases. Use Search API. Filter by stars/recency to avoid noise.
- **dev.to:** Search articles. Use Forem API. Filter by reactions/published date.
- **Stack Overflow:** Search questions. Use Stack Exchange API v2.1. Filter by score + answered status. Lower priority than GH+dev.to — gate behind explicit Stack Overflow intent if possible.

### Citation Policy
- All external sources must return a URL in the tool response. The agent prompt must instruct citing the URL in any response surfacing this content. Trust is non-negotiable in a community context.
- Owned content (blog/YT) returns canonical URLs (blog.codewithahsan.dev/{slug}, youtube.com/watch?v={id}).
- Third-party content returns upstream URL plus source label (e.g., "via dev.to").

### Credentials & Env Vars
- `GHOST_CONTENT_API_KEY` (already may exist for /api/blog rewrite — verify)
- `YOUTUBE_API_KEY` (may exist from v4.0 admin course creator — verify; reuse if so)
- `GITHUB_TOKEN` (optional; unauth fallback)
- Stack Overflow + dev.to need no auth for basic search
- All env vars: use `printf '%s'` when piping to Vercel/GH per past incident — no trailing newlines

### Testing
- Python unit tests under `agent/tests/` for each new tool function (mock httpx responses, matches existing test pattern).
- Vitest tests under `src/app/api/content/**/__tests__/` for the new Next.js API routes (mock upstream fetch, verify cache headers + response shape).
- Update root agent tests if the agent.py routing rules are tested.

### Claude's Discretion
- Exact tool function signatures and return-dict shapes (follow existing conventions in `platform_client.py` and `sub_agents/*.py`).
- Result count limits per query (suggest: 5 results max per source, ranked).
- Whether to add a unified `search_all_external` tool or keep tools split by source (default: split — clearer for the LLM).
- Whether to embed full blog post content or excerpts (default: excerpts in search; agent calls a separate "fetch full post" tool when needed).
- Plan task breakdown (waves, dependencies) — defer to planner.

</decisions>

<specifics>
## Specific Ideas

- Three plans likely (planner to validate):
  - Plan 01: Ghost blog — Next.js API route + ISR cache + `content_agent.search_blog_posts` tool + tests
  - Plan 02: YouTube — Next.js API route + ISR cache + `content_agent.search_youtube_videos` tool + tests. Verify reuse of v4.0 YT helper.
  - Plan 03: `external_knowledge_agent` with three tools (GitHub, dev.to, Stack Overflow) — direct httpx, no Next.js wrapper. Plus root agent routing updates + ADK redeploy.
- Optionally consolidate blog + YT into a single plan if they share infrastructure heavily.
- Ghost Content API: `https://blog.codewithahsan.dev/ghost/api/content/posts/?key={key}&limit=5&filter=title:~'{q}'` (verify exact param syntax in planner).
- YouTube Data API v3 search: `https://www.googleapis.com/youtube/v3/search?key={key}&q={q}&type=video&channelId={CHANNEL_ID}&maxResults=5` — confirm channel ID via existing v4.0 code.
- GitHub search: `https://api.github.com/search/repositories?q={q}&sort=stars&order=desc`
- dev.to: `https://dev.to/api/articles?per_page=5&top=7&tag={q}` (or `?search=` if available)
- Stack Overflow: `https://api.stackexchange.com/2.3/search/advanced?order=desc&sort=votes&q={q}&site=stackoverflow&filter=withbody`

</specifics>
