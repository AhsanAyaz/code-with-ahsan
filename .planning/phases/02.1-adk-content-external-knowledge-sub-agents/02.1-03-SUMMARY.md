# Phase 02.1 Plan 03: external_knowledge_agent + Cloud Run Deploy Summary

external_knowledge_agent sub-agent with GitHub, dev.to, and Stack Overflow direct httpx tools — wired into root agent as fallback after specialized sub-agents.

## What Was Built

**Task 1 — external_knowledge_agent sub-agent**

- `agent/community_assistant/sub_agents/external_knowledge_agent.py`: Three module-level `httpx.Client` instances (one per upstream API). Three shape helpers (`_shape_github_repo`, `_shape_devto_article`, `_shape_stackoverflow_question`) mapping upstream JSON to LLM-friendly dicts with canonical `url` fields. Three tool functions:
  - `search_github_repos(query)`: Calls `api.github.com/search/repositories` sorted by stars. Optional `Authorization: Bearer {GITHUB_TOKEN}` header when env var is set.
  - `search_devto_articles(query)`: Calls `dev.to/api/articles` with `tag=` param; fallback to `top=7` with client-side filter when tag returns empty and query has spaces.
  - `search_stackoverflow_questions(query)`: Calls `api.stackexchange.com/2.3/search/advanced` with `answers=1` (answered-only). `html.unescape()` applied to titles.
  - `external_knowledge_agent = LlmAgent(...)` with citation mandate and FALLBACK labeling.

- `agent/tests/fixtures/external_github.json`: 2 realistic GitHub repos (angular/angular, ngrx/platform) with all `_shape_github_repo` fields.
- `agent/tests/fixtures/external_devto.json`: 2 realistic dev.to articles with all `_shape_devto_article` fields.
- `agent/tests/fixtures/external_stackoverflow.json`: 2 SO questions; first title contains `&quot;` HTML entity to exercise `html.unescape`.

- `agent/tests/test_external_knowledge_agent.py`: 10 pytest tests:
  - 3 happy-path (URL domain assertions per source)
  - 3 error-path (monkeypatch raises `httpx.ConnectError`, asserts error dict with empty items list)
  - 4 edge cases (auth header present, auth header absent, HTML entity unescaping, empty query → all three tools return error)

**Task 2 — Root agent wiring**

## Decisions Made

1. **Direct httpx vs. Next.js proxy**: Locked in CONTEXT.md as D-CONTEXT-caching-split. GitHub (60/hr unauth, 5000/hr with token), dev.to (~1000/hr), SO (~10k/day) quotas are generous for the demo. No ISR caching benefit applies to third-party upstreams — every query hits the upstream directly.
2. **GITHUB_TOKEN optional**: Unauthenticated quota (60/hr) is sufficient for the community Discord demo. Users can create a `github-token` Secret Manager entry and add it to the Cloud Run deploy `--set-secrets` flag for the higher quota.
3. **Three dedicated httpx.Client instances**: Per-client form (`_GITHUB_CLIENT`, `_DEVTO_CLIENT`, `_STACKOVERFLOW_CLIENT`) documents upstreams and enables independent test monkeypatching.
4. **FALLBACK labeling**: Both agent description and ROOT_INSTRUCTION delegation bullet explicitly mark external_knowledge_agent as fallback — routing precedence rule added to the root agent's Rules block.
5. **html.unescape for SO titles**: Stack Overflow API returns HTML-encoded titles (e.g., `&quot;`). `html.unescape()` in `_shape_stackoverflow_question` fixes this before LLM sees the text.

## GITHUB_TOKEN Secret Manager Note

If `GITHUB_TOKEN` is added to Secret Manager before deploy, use `printf '%s'` (NOT `echo`) per past env-var hygiene incident — `echo` adds a trailing newline that corrupts the secret:

```bash
printf '%s' "$GITHUB_TOKEN" | gcloud secrets create github-token --data-file=- --replication-policy=automatic
# or add a new version:
printf '%s' "$GITHUB_TOKEN" | gcloud secrets versions add github-token --data-file=-
```

Then extend `--set-secrets` flag: `...,GITHUB_TOKEN=github-token:latest`.
