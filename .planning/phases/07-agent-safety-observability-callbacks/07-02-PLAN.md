**Short-circuit semantics:**
- `before_tool_cache`: return `dict` → skip the actual tool call, use that dict as the result. Return `None` → let the tool run normally.
- `after_tool_cache`: return `None` → pass through original `tool_response` unchanged. (RESEARCH §11 example_05 returns `None` to keep the original response intact while writing it to the cache.)

**Async note:** Both callbacks declared `async def` per RESEARCH §Summary line 16 — wall-clock reads (`datetime.now`) are non-blocking inside the Discord asyncio event loop. ADK awaits via `inspect.isawaitable(result)`.

**Cache key derivation (RESEARCH §5 + §9 Pitfall 3):**
```python
def _derive_cache_key(tool_name: str, args: dict) -> str:
    q = str(args.get("query", "")).lower().strip()
    q_norm = re.sub(r"[^\w\s]", "", q).strip()
    return f"{tool_name}__{q_norm}"
```
- `tool_name` prefix prevents `search_blog_posts(query='angular')` and `search_youtube_videos(query='angular')` from sharing a key.
- Lowercase + strip + punctuation removal normalizes `"Angular Signals!"`, `"angular signals"`, and `"  angular signals  "` to the same key.
- Empty query → key = `f"{tool_name}__"` (still a valid key; if user-input flow ever calls with empty query, the cache still works coherently — though the upstream tool itself returns an error for empty query per existing content_agent.py L57).

**TTL mechanism (RESEARCH §5):**
- Constants: `CACHE_TTL_SECONDS = 600` (10 minutes).
- Cache value shape: `(iso_timestamp_str, result_dict)` — a 2-tuple of `datetime.now(timezone.utc).isoformat()` and a deep-copied response dict.
- Freshness check: `(now - datetime.fromisoformat(ts_str)).total_seconds() < CACHE_TTL_SECONDS`.
- Wall-clock based (RESEARCH §5 + §15 A3 — Cloud Run uses NTP, no skew risk at single-instance scale).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM-generated `args` dict → `before_tool_cache` | The LLM populates the `query` argument from user content. Plan 07-01's PII sanitizer already redacted the user content before the LLM saw it, so `args["query"]` cannot contain raw PII. Cache keys may still contain LLM-generated arbitrary text. |
| `tool_context.state["app_cache"]` → cross-turn state | Cache entries live across turns within a session. A cached response from one turn could be re-served on a later turn for the same query — by design. Privacy boundary: no PII present (sanitized upstream) AND no cross-user leakage (each Discord user has a separate session). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-08 | Tampering | Cache key collision across tools | mitigate | Key derivation prefixes `tool_name` (`f"{tool_name}__{normalized_query}"`), preventing `search_blog_posts(query='angular')` from sharing a key with `search_youtube_videos(query='angular')`. Test 7 (`test_tool_name_isolation`) is the regression sentinel. |
| T-07-09 | Information Disclosure | Cached PII bleed-through | mitigate | Plan 07-01's `before_model_callback` redacts user content BEFORE the LLM sees it; the LLM's `query` argument therefore cannot contain raw PII. If a future bug short-circuits the PII layer, cache entries would NOT contain raw user PII because the tool's `query` arg comes from the LLM (which only saw the redacted text). Defense-in-depth: cache key normalization strips punctuation but keeps Discord-snowflake digits — review if any cache events show 17-19 digit substrings post-deploy. |
| T-07-10 | Tampering | Cache poisoning via error response | mitigate | `after_tool_cache` early-returns `None` when `tool_response.get("status") == "error"`. Test 3 (`test_after_tool_cache_does_not_cache_error_response`) is the regression sentinel. A transient Ghost / YouTube 5xx therefore cannot poison the cache for 10 minutes. |
| T-07-11 | Denial of Service | Unbounded cache growth | accept | Cache lives in `InMemorySessionService` session state, scoped per Discord user, surviving only until Cloud Run restart (which happens on every redeploy + on scale-down). Typical conversation length is <10 distinct queries → cache size <10 entries. No LRU needed for v7.0; Phase 8 Firestore session migration is the natural eviction point.
