# Quick Task 260609-ep2: Fix Monthly Challenge Bugs

Three bugs reported on the recently shipped Monthly Challenge feature. Root causes
found by reading the code. All fixes are surgical.

## Bug 1 — "Failed to submit project" (500)

**Root cause:** `createSubmission` in `src/services/ChallengeService.ts` strips
`userAvatar` when undefined before writing to Firestore, but **not** `demoUrl`.
When a user leaves the optional Demo URL blank, the submission route passes
`demoUrl: undefined`, which gets written via `transaction.set` → Firestore Admin
SDK rejects `undefined` values → 500 "Failed to submit project".

(`firebaseAdmin.ts` does not set `ignoreUndefinedProperties` — known project
constraint.)

**Fix:** Delete `demoUrl` from the write payload when undefined, mirroring the
existing `userAvatar` handling.

- File: `src/services/ChallengeService.ts`

## Bug 2 — "Participate" button does not reflect joined state

**Root cause:** Neither the inline button in `ChallengeCard.tsx` (the one in the
screenshot, on `/challenges`) nor the `ParticipateButton.tsx` component (challenge
detail page) checks whether the current user has already joined. They always show
an enabled "Participate".

**Fix:** On mount, when a user is logged in, fetch `GET /api/challenges/{id}/participants`
(`{ joined }`) and, if joined, render the button disabled with label "Joined".
After a successful join, flip local state to joined.

- File: `src/components/challenges/ChallengeCard.tsx`
- File: `src/components/challenges/ParticipateButton.tsx`

## Bug 3 — Start date off by one + edits "not persisting"

**Root cause:** `dateInputToIso` in `src/components/admin/ChallengeForm.tsx` builds
`new Date("YYYY-MM-DDT00:00:00.000")` with **no timezone** → parsed in the admin's
local timezone, then `.toISOString()` converts to UTC. For an admin ahead of UTC
(e.g. PKT, UTC+5), local midnight Jun 1 becomes `2026-05-31T19:00:00Z`. The stored
date's calendar day is now May 31. Edit prefill (`.split("T")[0]`) and the
date-range display then show May 31, and re-saving re-applies the same shift, so it
looks like edits never persist.

Display helpers compound it: `formatChallengeDateRange` (`src/lib/challenges.ts`)
and the admin list (`src/app/admin/challenges/page.tsx`) format without
`timeZone: "UTC"`, so even a correctly-stored date can render off-by-one for
viewers not at UTC.

**Fix:**
1. `dateInputToIso`: parse as UTC by appending `Z`, so the stored ISO's date
   component always equals the typed date.
2. `formatChallengeDateRange`: add `timeZone: "UTC"` to the formatter.
3. Admin list `toLocaleDateString`: pass `{ timeZone: "UTC" }`.

- File: `src/components/admin/ChallengeForm.tsx`
- File: `src/lib/challenges.ts`
- File: `src/app/admin/challenges/page.tsx`

**Note:** The existing June 2026 challenge already has the shifted date stored.
After this fix the admin must re-save it once (edit → save) to correct the value;
the display fix alone will not retro-correct already-bad data.
