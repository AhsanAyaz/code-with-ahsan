# VIS-57 — Disable "Book a Session" for non-mentees + email mentor on booking

GitHub: https://github.com/AhsanAyaz/code-with-ahsan/issues/206

## Background / data model

- An **active mentee of a mentor** = a doc in Firestore collection `mentorship_sessions`
  where `mentorId == <mentor>` AND `menteeId == <viewer>` AND `status == "active"`.
  This is the same query `findMentorshipSessionInfo()` already uses in the bookings route.
- On the mentor public profile, `MentorProfileClient` already computes `requestStatus`
  from `GET /api/mentorship/mentee-requests?menteeId=` (which reads `mentorship_sessions`).
  `requestStatus === "active"` is exactly "viewer is an active mentee of this mentor".

## AC

- Non-mentees see the Book button disabled/hidden with a tooltip explaining why.
- Active mentees can book a session normally.
- Mentor receives an email notification on booking.

## Task 1 — Gate the Book-a-Session controls on the mentor profile (client)

**files:** `src/app/mentorship/mentors/[username]/MentorProfileClient.tsx`

**action:**
- Add a derived `const canBook = requestStatus === "active";` near the render
  (after `requestStatus` state is available).
- The two existing "Book a Session" entry points both link to `/mentorship/book/${mentor.uid}`:
  1. Inside the Availability card (the `mentor.hasTimeSlots` block, ~line 791).
  2. The standalone "Book a Session" card (`mentor.hasTimeSlots && !availableDays.length`, ~line 819).
- For each: when `canBook` is true, keep the existing `<Link>`. When false, render a
  **disabled** button instead (DaisyUI `btn btn-primary btn-disabled` / `disabled`) wrapped
  in a tooltip (`<span className="tooltip" data-tip="...">`) with copy like:
  "Only active mentees of this mentor can book a session. Request mentorship first."
- Keep markup/styling consistent with the surrounding DaisyUI components. Do not remove the
  cards — just swap the actionable link for the disabled+tooltip variant when `!canBook`.

**verify:** `npm run lint` (or `npx tsc --noEmit`) passes; visually the booking link is a
disabled button with tooltip when `requestStatus !== "active"`.

**done:** Non-active-mentees see a disabled Book control with tooltip; active mentees see the working link.

## Task 2 — Guard the booking page (client) + server enforcement

**files:** `src/app/mentorship/book/[mentorId]/page.tsx`, `src/app/api/mentorship/bookings/route.ts`

**action (page):**
- After the existing `user`/`self` guards, fetch whether the viewer is an active mentee of
  this mentor. Reuse the existing endpoint: `GET /api/mentorship/mentee-requests?menteeId=${user.uid}`,
  find a request with `mentorId === mentorId` and `status === "active"`. Track loading.
- If not an active mentee, render a friendly blocking state instead of `<TimeSlotPicker>`:
  heading "You're not an active mentee of this mentor", short explanation, and a
  `Link` to the mentor profile / browse mentors. Show the spinner while this check loads.

**action (bookings POST route):**
- In `POST`, after fetching the mentee profile and before creating the booking, verify an
  active mentorship session exists between `mentorId` and `auth.uid`:
  query `mentorship_sessions` where `mentorId == mentorId`, `menteeId == auth.uid`,
  `status == "active"`, `limit(1)`. If empty, return
  `NextResponse.json({ error: "You must be an active mentee of this mentor to book a session." }, { status: 403 })`.
  (Factor into a small helper or reuse the shape of `findMentorshipSessionInfo`.)

**verify:** typecheck/lint passes; a non-active-mentee POST returns 403; an active mentee still books.

**done:** Booking page blocks non-active-mentees; API rejects non-active-mentee bookings with 403.

## Task 3 — Email the mentor on booking

**files:** `src/lib/email.ts`, `src/app/api/mentorship/bookings/route.ts`

**action (email.ts):**
- Add `export async function sendBookingConfirmationEmail(...)` following the existing
  patterns (uses `sendEmail`, `wrapEmailHtml`, the shared `emailStyles`). Params should
  carry: mentor `{ displayName, email }`, mentee display name, the formatted date/time
  string, timezone abbr, duration (30 min), optional session-type/template label, and a
  `pendingApproval: boolean` flag.
  - When `pendingApproval` is false: subject like
    `📅 New Session Booked by <mentee>` — body confirms the booked session details and links to dashboard.
  - When `pendingApproval` is true: subject like
    `📋 Booking Approval Requested by <mentee>` — body asks the mentor to approve/decline from the dashboard.

**action (bookings POST route):**
- After the booking is created and the existing Discord notification block, send the email
  to the mentor **non-blocking** (mirror the Discord `.catch(...)` style — never throw / never
  block the response). Pull the mentor email from `mentorData.email` (mentor profile doc).
  Format date/time in the mentor's timezone (reuse the `mentorTimezone` / `formattedDate` /
  `formattedTime` / `tzAbbr` values already computed in the route; if the Discord block is
  skipped because Discord isn't configured, compute the same values for the email path).
  Pass `pendingApproval: bookingStatus === "pending_approval"`.
- Import `sendBookingConfirmationEmail` at the top of the route.

**verify:** typecheck/lint passes; email send is wrapped so a Resend failure does not break booking.

**done:** Mentor gets an email on both confirmed and pending_approval bookings; failures are non-blocking.
