# Quick Task 260521-h86: Sitemap SEO Cleanup Summary

Migrated the hand-curated 116-URL `public/sitemap.xml` (which contained 18 auth-gated URLs contradicting `public/robots.txt`) to a dynamic Next.js 16 `MetadataRoute.Sitemap` at `src/app/sitemap.ts` that reads from the same content provider the routes use — now emits 270 URLs with zero robots-disallowed paths.

## Outcome

- `/sitemap.xml` is now served by `src/app/sitemap.ts` (a single default-exported async function returning `MetadataRoute.Sitemap`).
- The static `public/sitemap.xml` is deleted, and the `postbuild` step that resurrected it on every build has been removed.
- The sitemap is consistent with `public/robots.txt`: every previously emitted auth-gated URL is gone, every public route is present.

## Surprises observed

**Event slugs were renamed since the old sitemap.** The old sitemap had no per-event slugs — the dynamic event route `/events/[event-slug]` did exist but wasn't crawled. As of commit `595023d` ("fix(events): improve SEO for all event slugs, titles, descriptions"), the event slugs in `src/content/events.generated.json` use SEO-friendly forms:

- `BWAI-May2026` → `build-ai-travel-agent-adk-gemini-cli-may-2026`
- `cwa-promptathon-2026` → `cwa-promptathon-generative-ai-hackathon-2026`
- `hackstack-2023` → `hackstack-pakistan-fullstack-hackathon-2023`

The dynamic sitemap picks these up automatically via `getEvents()`. The hardcoded routes `/events/cwa-promptathon/2026`, `/events/cwa-promptathon/2026/sponsorship`, and `/events/hackstack/2023` are emitted separately in the static list (they are different file-system routes, not slug-based dynamic routes).

**`CourseContent` and `PostContent` have no `updatedAt` field**, despite the plan's interface block listing it. Inspected `src/types/content.ts` — only `publishedAt` exists. Resolved by using `publishedAt` with a `now()` fallback, which is enough to keep `lastmod` non-null for every entry.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Critical functionality] Removed `postbuild` script + `scripts/generate-sitemap.js`**
- **Found during:** Task 1 (file inventory before commit).
- **Issue:** `package.json` declared `"postbuild": "node scripts/generate-sitemap.js"`, which writes `public/sitemap.xml` on every build. The plan's must-have truth is "Stale public/sitemap.xml is removed so Next.js dynamic sitemap is the single source of truth" — leaving `postbuild` in place would have resurrected the static file after every build and shadowed `/sitemap.xml`, defeating the entire migration.
- **Fix:** Removed the `postbuild` line from `package.json` and deleted the now-orphaned `scripts/generate-sitemap.js`. The script's logic is fully superseded by the dynamic `MetadataRoute.Sitemap` and is no longer referenced anywhere.
- **Files modified:** `package.json` (1 line removed), `scripts/generate-sitemap.js` (deleted).
- **Commit:** `853b826`

### Plan field correction (no code change)

**2. `CourseContent` interface drift between plan and actual types**
- **Plan stated:** `CourseContent` has `updatedAt?: string | null`.
- **Actual:** `src/types/content.ts` has only `publishedAt?: string | null` on `CourseContent` (no `updatedAt`). Same for `PostContent`.
- **Action:** Used `publishedAt` as `lastModified` source with `now()` fallback. Documented in code comments. No type cast or unsafe access required.

### Worktree environment fixes (out of scope for commit)

**3. [Worktree-only, not committed] `npm install` to create `node_modules` in the worktree.**
- The Claude Code worktree had no `node_modules`. `next.config.ts` sets `turbopack.root: ".."` which resolves to `.claude/worktrees/` from inside a worktree — Turbopack then can't find `next/package.json`. Symlinking the main repo's `node_modules` was attempted first but Turbopack panicked ("Symlink invalid, points out of filesystem root"). Resolved by running `npm install` in the worktree (gitignored side effect, not part of the commit). See `.planning/quick/260521-h86-sitemap-seo-cleanup/deferred-items.md` for details.
